# PageIndex Search Skill

Answer questions by **navigating a PageIndex JSON tree** and extracting only the pages that matter.
**You (Claude) are the LLM agent.** No vector database or API key is required.

**Prerequisites:** A `*_structure.json` file already generated by the `pageindex` skill.
The page extraction script is at `./scripts/extract_pages.py` by look at the `pageindex` skill first.

---

## Core Concepts

### What is Tree Search?

A PageIndex JSON tree has nodes with `summary` / `prefix_summary`, page ranges (`start_index`, `end_index`), and children (`nodes`). Instead of scanning every page, you traverse the tree:

1. Read top-level node summaries to identify promising branches.
2. Drill into children of relevant branches.
3. Once at leaf nodes (or nodes with a tight enough page range), extract those pages.
4. Read the extracted text and answer the query.

This is fundamentally more efficient than full-text search: you read only the pages that the tree structure predicts are relevant.

### Single Document vs Multi-Document

| Scenario | Method |
|----------|--------|
| One document, one `_structure.json` | **LLM Tree Search** |
| Few documents, no structured metadata | **Description Search** → Tree Search |
| Many documents with metadata (dates, type, etc.) | **Metadata Search** → Tree Search |
| Many documents, diverse topics | **Semantic Search** → Tree Search |

## Workflow A — Single Document Search (LLM Tree Search)

Use this when the user asks a question about a specific PDF that has already been indexed.

### Step 1 — Load the PageIndex

Read the `*_structure.json` file for the document.
```
Read: /path/to/document_structure.json
```

### Step 2 — LLM Tree Traversal (your reasoning task)

Examine the tree and apply this logic:

```
For each top-level node:
  - Read its title + summary/prefix_summary
  - Ask: "Could this node contain the answer to the query?"
  - If YES or MAYBE → add to candidate set
  - If the node has children → examine children the same way
  - Stop drilling when:
    (a) you reach a leaf node, OR
    (b) the node's page range is ≤ 5 pages (small enough to read directly)
```

**Selection prompt (use internally):**
> "Given query: '{query}'\n\nReview each node's title and summary. List node_ids whose content likely contains the answer. Be inclusive — it's better to retrieve a slightly broader set than to miss the answer."

Collect the final list of `node_id` values → convert to page ranges using `start_index` / `end_index`.

**Merge overlapping page ranges** to avoid re-reading the same pages.

### Step 3 — Extract Selected Pages

```bash
python ~/.claude/skills/pageindex/scripts/extract_pages.py \
  --pdf "/absolute/path/to/document.pdf" \
  --start <start_page> --end <end_page> \
  --output "/tmp/pageindex_search_result.json"
```

Run one extraction per distinct page range. Read each output file.

### Step 4 — Synthesize Answer

Read the extracted page text and answer the user's query directly. Cite the section title and page number(s) where relevant.

If the extracted text is insufficient, go back to Step 2 and expand the candidate set to sibling or parent nodes.

## Workflow B — Multi-Document: Description Search

Use when you have **a small collection** (< ~20 documents) and no structured metadata.

### Step 1 — Generate Descriptions

For each document's `_structure.json`, generate a one-sentence description:

> "Summarize this document in one sentence that distinguishes it from other documents in the collection. Base your description on the top-level nodes and their summaries."

Store descriptions as: `{ "doc_id": "path/to/structure.json", "description": "..." }`

### Step 2 — LLM Document Selection

Present all descriptions to an LLM (yourself) with the user query:

> "Query: '{query}'\n\nDocuments:\n{list of descriptions}\n\nWhich documents may contain information relevant to answering this query? Return the doc_ids."

### Step 3 — Tree Search on Selected Documents

For each selected document, run **Workflow A** above.

### Step 4 — Synthesize Across Documents

Combine retrieved passages and answer the query, noting which document each piece of information came from.

## Workflow C — Multi-Document: Semantic Search

Use when you have **many documents** covering diverse topics.

### Scoring Formula

```
DocScore = (1 / sqrt(N + 1)) * sum(ChunkScore for all chunks of that doc)
```

- N = number of chunks from that document that appeared in top-K results
- ChunkScore = cosine similarity of the chunk to the query
- The sqrt denominator gives diminishing returns as N grows — favors fewer, highly-relevant chunks over many weakly-relevant ones

### Pipeline

1. **Chunk & embed** all documents (chunk each node's text into ~256-token segments; embed using any embedding model).
2. **Vector search**: retrieve top-K chunks for the query.
3. **Score documents**: apply the formula above to rank documents by DocScore.
4. **Select top documents** (top 3–5 by DocScore).
5. **Tree Search**: run **Workflow A** on each selected document.

> **Note:** You retrieve nodes (page ranges), not individual chunks. Chunks are only used for scoring.

---

## Workflow D — Multi-Document: Metadata Search

Use when documents have **clear categorical attributes** (e.g., financial reports, legal filings, medical records grouped by date/type/entity).

### Pipeline

1. Store each document's metadata + `doc_id` (path to its `_structure.json`) in a database or dictionary.
2. **Transform query → filter**: use LLM to extract filter criteria from the user query (e.g., "company=AAPL, year=2023, type=10-K").
3. **Filter documents**: apply filters to retrieve matching `doc_id` values.
4. **Tree Search**: run **Workflow A** on each matched document.

---

## Hybrid Tree Search (Advanced)

For large documents where LLM traversal alone is slow, combine two parallel strategies:

| Strategy | How |
|----------|-----|
| **Value-based (fast)** | Embed each node's text chunks. For the query, score nodes using `NodeScore = (1/sqrt(N+1)) * sum(ChunkScore)`. Take top-K nodes. |
| **LLM-based (accurate)** | Traverse the tree with LLM reasoning as in Workflow A. |

Run both **simultaneously** (or sequentially if parallelism is not available), merge their node lists (deduplicating), then extract and read the merged set. The LLM agent decides when sufficient information has been collected.

## Quick Decision Guide

```
User asks a question about a document
│
├─ Do you have a *_structure.json for it?
│   └─ YES → Workflow A (LLM Tree Search)
│
├─ Multiple documents?
│   ├─ < 20 docs, no metadata → Workflow B (Description Search)
│   ├─ Many docs, structured metadata → Workflow D (Metadata Search)
│   └─ Many docs, diverse topics → Workflow C (Semantic Search)
│
└─ Large document + speed matters → Hybrid Tree Search
```

## Step-by-Step Example (Single Document)

**User query:** "What distillation method is used to speed up inference?"

1. **Load** `2511.22699v3_structure.json`
2. **Scan top-level nodes** → Section 4 "Model Training" (`prefix_summary` mentions distillation) is a clear match
3. **Drill into Section 4's children** → node `0016` "4.5 Few-Step Distillation" (pages 20–22) + its children `0017`, `0018`, `0019` are the exact nodes
4. **Extract pages 20–22:**
   ```bash
   python ./scripts/extract_pages.py \
     --pdf "D:/vjump/fun-paper/2511.22699v3.pdf" \
     --start 20 --end 22 \
     --output "/tmp/search_result.json"
   ```
5. **Read** `/tmp/search_result.json` → answer with specific details about Decoupled DMD and DMDR

## Tips

- **Be inclusive in Step 2**: it is cheaper to read 2 extra pages than to miss the answer entirely.
- **Merge page ranges**: if nodes `0014` (pages 18–19) and `0015` (pages 19–20) are both selected, extract pages 18–20 in one call.
- **Parent fallback**: if a leaf node's text is insufficient, extract the parent node's full range.
- **Re-ranking**: if multiple passages are found, rank them by relevance before presenting to the user.
- **Cite sources**: always tell the user which section and page number the answer came from.
